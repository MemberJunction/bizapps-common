# PostgreSQL Support

BizApps Common installs into a MemberJunction database on **either SQL Server or PostgreSQL**
(including AWS Aurora / RDS, Cloud SQL, and Azure Database for PostgreSQL), mirroring how
MemberJunction itself supports both platforms.

The canonical migrations are authored once in SQL Server T‑SQL under `migrations/`. A converted
PostgreSQL set lives under `migrations-pg/`. A single environment variable selects which set runs.

---

## TL;DR — install on PostgreSQL

```bash
# 1. point at a PostgreSQL MJ instance
export DB_PLATFORM=postgresql
# DB_HOST/DB_PORT(5432)/DB_DATABASE/DB_USERNAME/DB_PASSWORD + CODEGEN_DB_* as usual

# 2. three-stage install (same shape as MemberJunction core on PG)
npm run mj:migrate            # DDL — runs migrations-pg/ automatically (folder auto-swap)
npm run mj:codegen            # regenerates CRUD functions, views, triggers as native PG
npx mj sync push --dir ./metadata   # seeds reference data (AddressTypes, etc.)

# 3. build + start
npm run build
npm run start
```

On SQL Server the flow is identical without `DB_PLATFORM` (defaults to `sqlserver`) and runs
`migrations/`.

---

## How platform selection works (inherited from the MJ CLI — no custom code)

The mechanism is entirely provided by `@memberjunction/cli` (pinned at 5.37.0). BizApps Common does
**not** reimplement any of it:

- **`DB_PLATFORM`** (`sqlserver` | `postgresql`) is read by the CLI. There is no new config key to set
  — do **not** hardcode `dbPlatform` in `mj.config.cjs`, because the file value would override the
  env var and defeat per-environment switching.
- **`mj migrate`** swaps the migrations folder `migrations` → `migrations-pg` automatically when
  `DB_PLATFORM=postgresql`, and dispatches to the PostgreSQL Skyway provider. The `.pg.sql` /
  `.pg-only.sql` suffix is cosmetic — dialect selection is purely by folder.
- The **runtime** (MJAPI) selects `@memberjunction/postgresql-dataprovider` from the same
  `DB_PLATFORM` env var. That provider ships transitively with `@memberjunction/server`, so no extra
  runtime dependency is required.

`npm run mj:migrate` passes `--schema __mj_BizAppsCommon` so the app's `${flyway:defaultSchema}`
placeholder resolves to the app schema (and `${mjSchema}` → core `__mj`), exactly as the OpenApp
installer does.

## Keeping `migrations-pg/` in sync — the two‑command process

When you add or change a SQL Server migration under `migrations/`, regenerate its PostgreSQL twin:

```bash
npm run mj:migrate:convert      # mj migrate convert — converts only NEW/missing migrations
node scripts/pg-finalize.mjs    # deterministic post-conversion fixups (idempotent)
```

1. **`mj migrate convert`** (MemberJunction's existing rule-based SQL converter) does the bulk
   SS→PG translation and writes `migrations-pg/*.pg.sql`. It **skips** any migration that already has
   a `.pg.sql` / `.pg-only.sql` twin, so previously-shipped (and finalized) migrations are never
   rewritten.
2. **`scripts/pg-finalize.mjs`** applies a small set of deterministic, **idempotent** patches for the
   handful of patterns the converter does not fully handle. Re-running it on already-finalized files
   is a no-op. This is the "things we fix after the converter" layer — it is *code*, not hand-editing,
   so the process stays repeatable for every future migration.

If a brand-new migration uses a construct none of the existing rules cover, either add one rule to
`scripts/pg-finalize.mjs` (preferred — it then applies forever) or hand-author a `*.pg-only.sql`
supplement (MemberJunction's sanctioned manual-override mechanism for migrations the converter
cannot produce).

### What `pg-finalize.mjs` fixes (and why)

| # | Patch | Converter gap it compensates for |
|---|-------|----------------------------------|
| 1 | Quote FK-join alias references | Converter quotes the `AS "alias"` definition but leaves references unquoted → PG case-folds them → "missing FROM-clause entry". |
| 2 | Neutralize skipped triggers / unrecognized batches | Converter comments only the `CREATE TRIGGER` line (or marks a batch `UNKNOWN`) but leaves invalid T‑SQL bodies (`@@ROWCOUNT` guards, truncated triggers) live. These are regenerated by `mj codegen` on PG. |
| 3 | Cast boolean INSERT values | Converter emits native `TRUE/FALSE` for bizapps tables it sized, but `0/1` for core `__mj` tables it can't see. (Map in `scripts/pg-core-boolean-columns.json`.) |
| 4 | Cast boolean comparisons | `"boolCol" = 0/1` → `= FALSE/TRUE` (PG has no implicit boolean=integer operator). |
| 5 | Quote core object refs | Unquoted mixed-case `__mj.vwGeneratedCodeCategories` → folds to a non-existent lowercased relation. |
| 6 | Drop-before-create views | `CREATE OR REPLACE VIEW` cannot rename/reorder columns; a regeneration migration reshapes a view → 42P16. We DROP first so the create is fresh (`mj codegen` is the authoritative source for views). |
| 7 | Cast `_Clear` flag args | CodeGen "tolerant" CRUD sprocs take boolean `<field>_Clear` flags; the converter passes integer `:= 1/0`, which won't match the boolean parameter. |
| 8 | Defer bizapps sproc seed blocks | Metadata_Sync seeds type tables by calling `__mj_BizAppsCommon."spCreate*"`, which don't exist until `mj codegen` runs. The same seed data ships in `metadata/` and is loaded by `mj sync push` after codegen. |
| 9 | Normalize bizapps schema unquoted | Converter leaves the schema name unquoted (PG folds to lowercase `__mj_bizappscommon`); `mj codegen` and the runtime also emit the schema unquoted, so we keep it consistently unquoted to avoid "relation does not exist". |

(The numbering above is logical; see `scripts/pg-finalize.mjs` for the exact, commented rule set,
plus two narrow one-offs — the `fnOrganizationParentID_GetRootID` inline table-valued function and
the `Person.DisplayName` `PERSISTED` computed column — which have no analogue in any other migration.)

## Why three stages on PostgreSQL

MemberJunction's PostgreSQL model is **DDL via migrations → CRUD functions/views/triggers via
`mj codegen` → reference data via `mj sync push`**. The CodeGen-generated stored procedures, base
views, and update triggers embedded in the SQL Server migrations are *regenerated natively* on
PostgreSQL by `mj codegen` (as PL/pgSQL functions and PG triggers), not translated. That is why the
finalize step neutralizes those sections rather than attempting a faithful T‑SQL→PL/pgSQL port, and
why the type-table seed data is applied by `mj sync push` after codegen has created the CRUD
functions.

`advancedGeneration` is disabled in `mj.config.cjs` so the app installs without AI-provider
credentials; it is an optional enrichment (AI form layouts) that does not affect schema, CRUD, or
sync. Enable it in an environment that has AI model credentials.

## Testing the object model

`scripts/pg-objectmodel-test.mjs` (`npm run test:objectmodel:pg`) is a comprehensive functional
test of the BizApps Common object model on PostgreSQL — it exercises the write path (the
CodeGen-generated CRUD functions), the read path (FK-join base views), the polymorphic links
(Person/Organization), directional relationships, and the distinctive model features
(`Person.DisplayName` STORED generated column, the Organization root-parent recursive TVF, and the
`UQ_Person_LinkedUserID` 1:1 constraint), with full CRUD round-trips. It is self-cleaning (deletes
its own test rows) and exits non-zero on any failure. Requires a PG instance with the app installed
(migrate → codegen → sync). Currently 17/17 passing.

## Known limitations

- **`mj app install` (the OpenApp installer) is SQL-Server-only in MJ 5.37.0.** It reads
  `mj-app.json`'s `migrations.directory` literally and uses the SQL Server Skyway provider, with no
  platform folder-swap. Installing onto a PostgreSQL MJ instance therefore uses the developer flow
  above (`mj migrate` / `mj codegen` / `mj sync push`), which is fully platform-aware. Reaching PG
  through `mj app install` requires a MemberJunction-core enhancement to the installer (platform-aware
  provider + `migrations-pg` selection) and is tracked as a framework follow-up.
